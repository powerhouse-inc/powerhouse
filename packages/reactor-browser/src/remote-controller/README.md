# Remote Document Controller

A lightweight controller that wraps `PHDocumentController` with remote push/pull capabilities via GraphQL. It provides type-safe, chainable action methods with automatic sync to a reactor server.

## Overview

`RemoteDocumentController` composes a `PHDocumentController` and adds remote sync. Use it when you need to create or edit documents locally and sync them with a reactor server, without setting up a full `ReactorClient` (PGLite, job queue, etc.).

It works with any document model controller generated by `PHDocumentController.forDocumentModel()`.

## Quick Start

```typescript
import {
  createClient,
  RemoteDocumentController,
} from "@powerhousedao/reactor-browser";
import { DocumentModelController } from "document-model";

const client = createClient("https://switchboard.example.com/graphql");
```

### Create a new document

Omit `documentId` and pass a `parentIdentifier` (the drive to create in):

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "",
    mode: "batch",
    parentIdentifier: "my-drive",
  },
);

// Type-safe, chainable actions (types inferred from DocumentModelController)
controller
  .setName({ name: "My Document Model" })
  .setModelName({ name: "MyModel" })
  .setModelDescription({ description: "A sample model" });

console.log(controller.header.name); // "My Document Model"
console.log(controller.status.pendingActionCount); // 3

// Push all pending actions to the remote
const result = await controller.push();
console.log(result.actionCount); // 3
console.log(controller.status.documentId); // "abc-123" (assigned by server)
```

### Pull an existing document

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "batch",
  },
);

console.log(controller.header.name); // "My Document Model"
console.log(controller.status.connected); // true

// Modify and push
controller.setModelName({ name: "UpdatedModel" });
await controller.push();
```

### Wrap a local controller

Use `RemoteDocumentController.from` to add remote capabilities to an existing controller instance. Pre-existing actions on the local controller are not tracked (only new actions through the remote controller are tracked).

```typescript
const local = new DocumentModelController();
local.setName({ name: "Draft" });

const remote = RemoteDocumentController.from(local, {
  client,
  mode: "batch",
  parentIdentifier: "my-drive",
});

// Pre-existing state is preserved
console.log(remote.header.name); // "Draft"

// New actions are tracked
remote.setModelName({ name: "MyModel" });
await remote.push(); // creates on remote + pushes the SET_MODEL_NAME action
```

## Modes

### Batch mode

In batch mode, actions accumulate locally until you call `push()`:

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "batch",
  },
);

controller.setName({ name: "A" });
controller.setName({ name: "B" });
console.log(controller.status.pendingActionCount); // 2

// Both actions sent in a single MutateDocument call
await controller.push();
```

### Streaming mode

In streaming mode, actions are automatically pushed after each synchronous chain via `queueMicrotask`:

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "streaming",
  },
);

// These are coalesced into a single push (same microtask)
controller.setName({ name: "Updated" }).setModelName({ name: "NewName" });
// push happens automatically after the current microtask
```

If push fails in streaming mode, actions remain in the tracker for retry on the next action. Use `onPushError` to handle errors:

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "streaming",
    onPushError: (error) => console.error("Push failed:", error),
  },
);
```

## Signing

Pass an `ISigner` to sign actions before they are pushed. If you have a Renown instance, use its built-in signer:

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "batch",
    signer: renown.signer,
  },
);

controller.setName({ name: "Signed action" });
const result = await controller.push();

// result.operations contains the signed actions that were sent
const signature = result.operations[0].context?.signer?.signatures[0];
```

Each action is signed with the operation's `prevOpHash` before push. Signatures are preserved through the push/pull roundtrip.

## Conflict handling

When multiple controllers edit the same document concurrently, a push may conflict with changes made by another peer. By default no conflict check is performed (last-write-wins). Set the `onConflict` option to enable pull-before-push conflict detection.

### Reject strategy

Throws a `ConflictError` if the remote has new operations since the last pull. The caller can inspect the conflict and decide what to do.

```typescript
import {
  ConflictError,
  RemoteDocumentController,
} from "@powerhousedao/reactor-browser";

const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "batch",
    onConflict: "reject",
  },
);

controller.setName({ name: "My change" });

try {
  await controller.push();
} catch (err) {
  if (err instanceof ConflictError) {
    console.log(err.conflict.remoteOperations); // new remote ops
    console.log(err.conflict.localActions); // pending local actions
  }
}
```

### Rebase strategy

Automatically pulls the latest remote state, then re-applies local actions on top. The actions get fresh `prevOpHash` values computed against the updated state.

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "batch",
    onConflict: "rebase",
  },
);

controller.setName({ name: "My change" });
// If remote changed, this pulls first, re-applies setName, then pushes
await controller.push();
```

### Custom merge handler

Provide a function that receives conflict details and returns the actions to push. Throw from the handler to abort.

```typescript
const controller = await RemoteDocumentController.pull(
  DocumentModelController,
  {
    client,
    documentId: "abc-123",
    mode: "batch",
    onConflict: (conflict) => {
      // Inspect what changed remotely
      console.log(conflict.remoteOperations);
      console.log(conflict.localActions);

      // Return the actions you want to push
      return [
        {
          type: "SET_NAME",
          input: { name: "Merged result" },
          scope: "global",
          id: crypto.randomUUID(),
          timestampUtcMs: Date.now().toString(),
        },
      ];
    },
  },
);
```

## API Reference

### RemoteDocumentController

```typescript
class RemoteDocumentController<TController extends PHDocumentController<any>>
```

#### Static factories

| Method                                                    | Description                                                                     |
| --------------------------------------------------------- | ------------------------------------------------------------------------------- |
| `RemoteDocumentController.pull(ControllerClass, options)` | Pull a document from remote (or prepare for a new one) and return a controller. |
| `RemoteDocumentController.from(controller, options)`      | Wrap an existing local controller with remote capabilities.                     |

Both return `RemoteDocumentControllerWith<C>`, which includes the base class plus all action methods from the wrapped controller.

#### Getters

| Property     | Type                 | Description                                    |
| ------------ | -------------------- | ---------------------------------------------- |
| `header`     | `PHDocumentHeader`   | Document header (name, id, type, etc.)         |
| `state`      | `TState`             | Document state (type inferred from controller) |
| `operations` | `DocumentOperations` | Operations grouped by scope                    |
| `status`     | `SyncStatus`         | Current sync status                            |

#### Methods

| Method               | Returns                       | Description                                                  |
| -------------------- | ----------------------------- | ------------------------------------------------------------ |
| `push()`             | `Promise<PushResult>`         | Push pending actions to remote, then pull latest state.      |
| `pull()`             | `Promise<RemoteDocumentData>` | Pull latest state from remote, replacing local document.     |
| `delete(propagate?)` | `Promise<boolean>`            | Delete the document on the remote.                           |
| `onChange(listener)` | `() => void`                  | Register a change listener. Returns an unsubscribe function. |

Action methods from the wrapped controller are also available and return `this` for chaining.

### RemoteControllerOptions

```typescript
type RemoteControllerOptions = {
  client: ReactorGraphQLClient; // GraphQL client from createClient()
  documentId?: string; // Remote document ID (omit for new documents)
  mode: "streaming" | "batch"; // Push strategy
  signer?: ISigner; // Optional action signer
  branch?: string; // Branch name (default: "main")
  parentIdentifier?: string; // Parent drive ID (required for new documents)
  onConflict?: ConflictStrategy; // Conflict resolution strategy (default: no check)
  onPushError?: (error: unknown) => void; // Called when a streaming-mode push fails
};
```

### PushResult

```typescript
type PushResult = {
  remoteDocument: RemoteDocumentData; // Server response
  actionCount: number; // Number of actions pushed
  operations: ReadonlyArray<Action>; // The signed actions sent to the server
};
```

### SyncStatus

```typescript
type SyncStatus = {
  pendingActionCount: number; // Actions waiting to be pushed
  connected: boolean; // true if documentId is set
  documentId: string; // Remote document ID
  remoteRevision: Record<string, number>; // Revision per scope
};
```

## Architecture

### Push flow

1. Flush pending actions from the tracker.
2. If this is a new document (no `documentId`), create it on the remote via `CreateEmptyDocument`.
3. If `onConflict` is set, check if the remote has new operations since the last pull. Handle conflicts according to the strategy (reject, rebase, or custom merge).
4. If a signer is configured, sign each action with its `prevOpHash`.
5. Send all actions in a single `MutateDocument` call.
6. Pull remote state to reconcile (remote is source of truth).

### Pull flow

1. Fetch the document via `GetDocument`.
2. Fetch all operations via paginated `GetDocumentOperations`.
3. Reconstruct a `PHDocument` from the remote state and operations.
4. Replace the inner controller with a new instance built from the pulled document.
5. Clear the action tracker.

### Action interception

When you call an action method (e.g. `controller.setName(...)`):

1. Snapshot operation counts per scope before applying.
2. Delegate to the inner controller to apply the action locally.
3. Detect which scope received the new operation.
4. Record `prevOpHash` and `prevOpIndex` from the previous operation in that scope.
5. Track the action for push.
6. In streaming mode, schedule a push via `queueMicrotask`.

### Remote as source of truth

After every `push()`, the controller pulls the latest state from the remote. This ensures:

- Local state never diverges from remote.
- Server-injected operations (e.g. `UPGRADE_DOCUMENT`) are incorporated.
- The operation history stays in sync for correct `prevOpHash` computation.
