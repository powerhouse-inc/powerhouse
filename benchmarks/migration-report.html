tsgo + oxlint migration

Migrated the entire lint + format + typecheck stack from tsc / eslint / prettier / eslint-plugin-better-tailwindcss to the Rust + Go ports — tsgo, oxlint, oxfmt, and oxlint-tailwindcss — with near 100% rule parity. ESLint and Prettier (and 7 ESLint plugins) are fully removed from the dependency tree. The two custom rules (logger, cli-cold-path) live in lint-plugins/ using oxlint's native plugin API (definePlugin + defineRule + createOnce).

monorepo: powerhouse ~2268 TS files node 24 · pnpm 11 branch: worktree-migrate-tsgo-oxlint

IssueImpactStatus
Markdown / MDX oxfmt corrupts MDX JSX-comment syntax ({/* ... */} → {/_ ... _/}), breaking academy docs and other generated MD files. handled **/*.md and **/*.mdx in .oxfmtrc ignore list. Markdown formatting still runs via prettier if desired (not gated by CI).

oxlint-tailwindcss (native plugin) — adopted

IssueImpactStatus
pnpm exports field resolution pre-build The native plugin loads each workspace's design system by following @import chains through package.json#exports. This repo's design-system / vetra / connect / powerhouse-vetra-packages map ./style.css → ./dist/style.css, which doesn't exist until build:css runs. Without it the plugin falls into a slow retry path: 38s on apps/connect/src alone, ~84s on full repo. resolved pnpm lint:ox now runs pnpm build:css first (~4s, builds the four Tailwind dist files in parallel). Full lint+format then takes ~28s on full repo, with the native plugin enabled. Also added array-fallback exports ("./style.css": ["./dist/style.css", "./style.css"]) as a hint, though Node's exports resolution does not actually honor array fallback for missing files — the build:css step is the real fix.
Stricter than better-tailwindcss Native plugin surfaces 58 unknown-class errors vs ~44 from the previous setup, plus enforce-canonical / sort-order / variant-order warnings that better-tailwindcss did not emit. triage
Tailwind v3 support oxlint-tailwindcss targets Tailwind v4 exclusively. This repo uses v4 so not an issue here. not blocking

Migrating eslint-disable comments

Empirical findings — oxlint accepts most existing ESLint directive syntax verbatim. Only the React Compiler rules loaded via jsPlugins required a rename.

Disable comment form Honored by oxlint? Notes
// eslint-disable-next-line @typescript-eslint/X yes oxlint accepts the old @typescript-eslint/ prefix as a synonym for its own typescript/.
// eslint-disable-next-line typescript/X yes Native oxlint id.
// oxlint-disable-next-line ... yes oxlint's own directive prefix; interchangeable with eslint-disable.
// eslint-disable-next-line no-unused-vars yes Bare core-rule names work as-is.
// eslint-disable-next-line react-hooks/rules-of-hooks yes Native rule; same prefix.
// eslint-disable-next-line react-hooks/set-state-in-effect no The React Compiler rules from eslint-plugin-react-hooks v7 are loaded via jsPlugins with alias react-hooks-extra (oxlint reserves the react-hooks name for the native plugin). Disable comments must use the new prefix: react-hooks-extra/set-state-in-effect, react-hooks-extra/incompatible-library, react-hooks-extra/static-components.

Codemod

benchmarks/migrate-disable-comments.mjs is the codemod that handles the only rename actually required: 

react-hooks/set-state-in-effect    → react-hooks-extra/set-state-in-effect
react-hooks/incompatible-library   → react-hooks-extra/incompatible-library
react-hooks/static-components      → react-hooks-extra/static-components

Run with node benchmarks/migrate-disable-comments.mjs --dry to preview, then without --dry to apply. On this repo: 25 replacements across 17 files, which suppressed 23 previously-unmatched set-state-in-effect findings.

Unused disable directives

oxlint supports --report-unused-disable-directives. After the codemod, this repo has ~65 unused directives, almost all for @typescript-eslint/no-unsafe-* rules that the new oxlint config has explicitly disabled (matching the old ESLint config, which also had them disabled). They are now stale and can be removed in a follow-up pass — none are gating today.

Recommended rollout

  1. Now (this PR) — tsgo as advisory pre-check; oxlint + oxfmt as the single blocking lint+format gate; ESLint and Prettier fully removed.
  2. Next sprint — triage the 81 oxlint correctness errors (mostly real bugs: react-hooks/rules-of-hooks, no-unsafe-optional-chaining, unicorn/*), then flip no-useless-assignment from warn to error once it leaves nursery.
  3. Optional — sweep the ~65 stale eslint-disable directives surfaced by oxlint --report-unused-disable-directives (mostly @typescript-eslint/no-unsafe-* on rules already disabled in the new config).
  4. 2–3 months — when tsgo hits 1.0 and @docusaurus/tsconfig drops baseUrl, swap pnpm typecheck to tsgo and keep tsc only as the CI authoritative gate.
